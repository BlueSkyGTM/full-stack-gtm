# MLOps 03 — Drift Monitoring

## Hook It
A model that scored 0.93 F1 at deployment can silently degrade to 0.71 with no errors, no crashes, no logs. Drift is the term for input or output distributions shifting away from training conditions. This beat frames the failure mode: what drift looks like in production, why standard uptime monitoring cannot catch it, and the business cost of serving stale predictions. Brief narrative: a lead-scoring model trained on pre-recession data continues to run "correctly" while systematically misranking prospects.

## Ground It
Three drift categories with distinct mechanisms: **covariate drift** (input feature distribution shifts while P(y|X) stays constant), **concept drift** (P(y|X) itself changes), and **prediction drift** (model output distribution shifts, often a lagging indicator of one of the first two). Statistical tests that quantify distributional distance: Kolmogorov-Smirnov for continuous features, Population Stability Index (PSI) for binned comparisons, Jensen-Shannon divergence as a symmetric, bounded alternative to KL divergence. Reference window vs. analysis window semantics. Threshold selection as a cost-benefit decision, not a statistical absolute.

## Build It
Implement drift detection from scratch using NumPy and SciPy before introducing a dedicated library. Code block 1: KS-test sweep over synthetic shifted distributions, printing feature-level p-values and flagging drift at α = 0.01. Code block 2: PSI computation over binned reference vs. current data with explicit threshold tiers (PSI < 0.1 stable, 0.1–0.25 moderate, > 0.25 significant). Code block 3: Jensen-Shannon divergence matrix across all features with a heatmap-style terminal printout. Then introduce **Evidently** as a library that packages these tests, produces HTML reports, and integrates with pipeline schedulers. Show a minimal Evidently report generated from CSV files with observable console output confirming drift flags.

## Ship It
Production drift monitor as a scheduled job, not a dashboard. Architecture: reference dataset stored as a snapshot (Parquet in S3/GCS), current window pulled from feature store or data warehouse, comparison job runs on cron (Airflow, Prefect, or GitHub Actions). Alerting rules: drift flag → Slack webhook → incident channel. Retraining trigger policy: N consecutive drift flags on high-importance features → open retraining ticket with linked drift report. Edge case: seasonal drift (Q4 vs. Q1) is expected and should be handled with seasonal reference windows, not alerts. Anti-pattern: retraining on drift alone without checking whether drift actually degraded the business metric. Exercise hook (easy): write a cron expression for daily drift checks. Exercise hook (medium): implement a Slack alerting function that fires only when PSI exceeds threshold on 3+ features simultaneously. Exercise hook (hard): build a multi-window reference system that rotates seasonal baselines and logs which baseline was active at detection time.

## Extend It
**Multivariate drift**: univariate tests miss correlated feature drift where individual marginals look stable but the joint distribution shifts. Approaches: Mahalanobis distance, domain classifier (train a binary classifier to distinguish reference from current — AUC significantly above 0.5 indicates drift). **Adaptive thresholds**: fixed PSI cutoffs break when feature cardinality or skewness varies across the feature set; quantile-based thresholds calibrated on historical drift noise reduce false positives. **Drift vs. degradation**: drift is a distributional observation; degradation is a performance observation. They correlate but are not identical. A model can drift without degrading (benign drift) or degrade without detectable univariate drift (label shift, concept drift with stable inputs).
